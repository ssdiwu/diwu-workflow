# /ddemo — Demo 能力规格

## 适用场景

当一件事的结果不可预期（不是「会不会做」，而是「做了结果稳定吗」），先隔离验证，再集成进版本。

典型场景：
- **验证不确定性**：新技术/新方案的效果是否稳定可预期
- **吸收外部知识**：从开源项目、论文、竞品中发现某个能力（如 prompt 技巧、算法实现、架构模式），提取出来封装成团队标准的 Demo，验证后加入能力库

小修小改（加字段、修 bug、改配置）无需执行本命令。

## Step 1：获取输入

优先级：
1. 用户在参数中提供了 Demo 名称或描述 → 直接使用
2. glob 扫描 `.doc/prd/` 目录，读取 PRD 中的 Demo 需求清单，列出让用户选择
3. 否则询问：要验证什么不确定性？

## Step 1.5：两级门控（必须通过才能继续）

在生成规格之前，先过两道门：

### 第一级：产品类型判断

**判断问题**：当前项目的核心价值来自哪里？

```
业务编排型：核心价值来自流程正确、数据准确
  → 大部分工作无需 Demo，直接做

能力构建型：核心价值依赖技术特性验证
  → 核心价值部分需要 Demo

混合型：既有确定性流程，也有不可预期能力
  → 只对不可预期部分做 Demo
```

**比例参考**：通常产品中不确定性工作占 20-40%（因产品而异），Demo 覆盖的是关键少数而非全部。

**判断锚点**：

- 正例（业务编排型）：CRM 系统、订单管理、报表生成 → 流程确定，无需 Demo
- 正例（能力构建型）：AI 对话引擎、实时推荐系统、音视频处理 → 核心能力需验证
- 边界例（混合型）：电商平台的「智能客服」模块 → 商品展示/下单流程无需 Demo，对话理解能力需要 Demo

### 第二级：能力 vs 功能判断

**判断问题**：换一个产品，这个 Demo 还能用吗？

```
✅ 能用 → 这是能力，继续
   示例：「消息的增删改查」「SSE 流式推送」「Prompt 稳定输出引用标记」

❌ 不能用 → 这是功能，重新定义
   示例：「Hera 的聊天页面」「订单详情展示」
```

如果判断为「功能」，提示用户重新定义——把产品特定部分剥离，找底层能力：

- ❌「Hera 会议引用功能」→ ✅「Prompt 能否稳定输出引用标记格式」
- ❌「用户签到打卡」→ ✅「定时任务调度的可靠性」

**命名规则**：能力确认后，确定 `demo-{能力关键词}` 名称。名字体现能力，不体现产品或功能。

## Step 2：建立上下文

**2a. 读取现有上下文（先查积木库）**

按以下顺序读取，不全量扫描：

1. `.doc/demo/README.md` → **优先检查已有积木**：扫描索引表中与当前 Demo 能力相关的条目
   - 状态为 `passed` → 直接复用，不重新验证，跳到 Step 6 提示复用路径
   - 状态为 `pending` 且能力重叠 → 提示用户：是复用/合并还是新建？确认后再继续
   - 无匹配 → 继续后续步骤
2. `.doc/README.md` → 了解系统文档全貌，定向读取相关文件
3. `.doc/adr/README.md` → 确认有无被否决的方案
4. `.doc/prd/README.md` → 找到触发本 Demo 的原始 PRD（列格式：`| 文件 | 摘要 | demos | 状态 |`），精读其中关于本 Demo 的描述

以上 README 不存在时，才 glob 扫描对应目录。

**2b. 识别不确定性来源**

从四类来源中定位本 Demo 的不确定性类型（可多选）：

| 类型 | 判断依据 | 示例 |
|------|---------|------|
| 能力缺口 | 团队从没做过这件事 | 第一次接入 TTS API |
| 结果不可预期 | 做了但不知道效果如何 | Prompt 能否稳定输出指定格式 |
| 方法未验证 | 知道目标但路径不确定 | 运营 SOP 能否跑通 |
| 集成风险 | 各自能跑但组合未测过 | 三个模块集成后的数据流 |

## Step 3：生成 Demo 规格

### 文档结构

#### 标题

```
# demo-{kebab-case-name}

**{能力一句话定义——换项目也成立，不含产品名称}**
```

#### 能力定义

一句话说清这个 Demo 验证什么能力：
- 不含产品名、功能名
- 换一个项目读这句话，同样成立
- 正例：「验证 Prompt 指令能否让 LLM 稳定输出 [cite:id] 格式引用标记」
- 反例：「验证 Hera 会议引用功能的 Prompt 效果」

#### 不确定性来源与核心问题

```
来源类型：[从四类中选]
核心问题：[具体描述——不是「会不会做」，是「结果能否稳定可预期」]
```

> 正例：「同一条 instruction，在不同 transcript 上，LLM 输出结构是否一致？」
> 反例：「能否实现引用功能」（工程能力问题，不是不确定性）

#### Demo 粒度自检

生成规格前自检三项：

```
□ 一句话能说清这个 Demo 解决什么（说不清 → 太大了，拆）
□ 预估代码量 / 文档量在合理范围内（代码类 200-500 行）
□ 不依赖其他未完成的 Demo（或依赖已明确接口）
```

第一条不满足时，停下来拆 Demo，再继续。

#### 核心验证资产

**这一节是 Demo 规格的核心，不允许留空。**

根据不确定性类型选择：

- **LLM/Prompt 类**：Prompt v1 全文（可直接复制运行）+ 测试 case 列表（正常路径 × 2、边界 case × 2、异常输入 × 1）
- **集成/数据流类**：测试 case 矩阵（输入 → 预期输出 → 验证方法），至少包含 2 条异常路径（上游返回空、字段缺失、超时）
- **第三方 API 类**：对接测试脚本骨架 + 边界 case 列表（超时、鉴权失败、响应格式异常）
- **UI/交互类**：交互用例矩阵（操作序列 → 预期状态），包含空状态、加载失败、超时三种异常
- **运营/方法类**：小规模试跑的执行步骤 + 观测指标列表

> 最容易犯的错：核心验证资产里 90% 是数据库 DDL、API 契约、完整实现代码 → 这是集成阶段的工作，不属于 Demo 规格。Demo 只关心「怎么验证这个不确定性」。

#### 通过标准

| 指标 | 目标 |
|------|------|
| [可量化指标名] | [具体数字，如 > 90%] |

每条必须可量化，不允许「效果良好」等模糊表述。

#### 依赖关系

```
前置 Demo（本 Demo 依赖谁的输出）：[无 / demo-xxx，需要它确定的 XXX 格式]
后置 Demo（谁依赖本 Demo 的输出）：[无 / demo-yyy，需要本 Demo 验证的 ZZZ]
接口约定：[本 Demo 对外输出的格式/结构，供后置 Demo 使用]
```

#### 边界映射

列出所有识别到的边界场景，标注每条是「Demo 内」还是「Demo 外（原因）」。不允许只写「不覆盖 X」而不说明为什么，也不允许跳过此节。

#### 集成注意事项（提前识别）

Demo 验证通过后集成进版本时的潜在差异。提前写，避免「Demo 通过但版本里不工作」：

- 依赖版本：[Demo 使用的版本 vs 版本项目可能使用的版本]
- 配置差异：[Demo 用 localhost / 版本项目用环境变量等]
- 接口适配：[Demo 的函数签名是否需要调整]

## Step 4：创建 Demo 目录结构

在项目根目录创建（`demo/` 不存在则先创建）：

```
demo/demo-{name}/
├── context/
│   └── 需求.md    ← 从 PRD 或用户输入中提取与本 Demo 相关的内容，写入此文件
└── README.md      ← 按六问题模板预填（验证完成后再补充结果和踩坑）
```

**README.md 预填模板**（`[待验证后填写]` 处留空，等实际运行后补充）：

```markdown
# demo-{name}: {能力一句话定义}

## 解决什么问题
{从能力定义直接取}

## 核心方案
[待验证后填写]

## 验证结果
[待验证后填写]

## 已知限制
[待验证后填写]

## 集成注意事项
{从规格文档的「集成注意事项」章节复制}

## 怎么复用
[待验证后填写]

## 踩过的坑
[待验证后填写]
```

## Step 5：写入规格文档

路径：`.doc/demo/DEMO-{kebab-case-name}-spec.md`

`.doc/demo/` 不存在时先创建目录，同时创建 `.doc/demo/README.md`。

## Step 5.5：更新 `.doc/demo/README.md`

```markdown
# Demo 索引

| 文件 | 能力（一句话） | 通过标准摘要 | 状态 |
|------|-------------|------------|------|
| DEMO-prompt-citation-spec.md | Prompt 能否稳定输出引用标记格式 | 准确率 > 90%，幻觉率 < 5% | pending |
```

列格式定义见 `.doc/schema.md` §Demo。

**首次创建 README（兼容旧版本）**：
- 如果 `.doc/demo/` 目录存在但 README.md 不存在，先 glob 扫描 `DEMO-*.md` 文件
- 读取每个 Demo 的能力定义和通过标准，生成完整索引表
- 状态默认为 `pending`，再追加当前新建的 Demo

**日常维护**：新建 Demo → 追加一行；验证通过 → 修改状态为 `passed`。

## Step 6：写入后提示

1. 显示规格文档路径 + demo 目录路径
2. 确认核心验证资产是否完整
3. 若有依赖关系：提示执行顺序（前置 Demo 未完成时不要继续）
4. 若有多个 Demo：提示其余 Demo 名称，可逐项继续运行 `/ddemo`
5. 所有 Demo 验证通过后，使用 `/dtask` 拆解集成任务

## 不做的事

- 不分析「哪些事需要 Demo」（这是 `/dprd` 的职责）
- 不批量处理多个 Demo（每次只处理一个）
- 不写产品 UX 文档（用户痛点、场景演示、UI 线框图）
- 不写生产架构（数据库 DDL、多服务 API 契约、灰度策略）
- 不写完整实现代码（demo/src/ 是执行阶段填写的）
- 不写入 dtask.json（由 `/dtask` 负责）
