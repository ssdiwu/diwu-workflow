# /dprd — 产品需求文档

## 模式判定

- 用户输入 `/dprd full` → **完整模式**（论证链延伸到"怎么做、怎么验证"，积木池包含技术方案、数据模型、验证方案等）
- 用户输入 `/dprd` → **产品模式**（论证链聚焦于"做什么、为什么"，积木池限于痛点、方案、规格、场景、UI）

## Step 1：确认核心定位

无论上下文是否已有描述，先确认核心定位再继续。

- 若用户**尚未描述**，直接问以下三个问题（一次问完）：
  - 要解决什么问题？
  - 目标用户是谁？
  - 核心场景是什么？
- 若用户**已有描述**，复述核心定位并请用户确认，确认后进入 Step 2。

## Step 2：澄清需求

**询问规则：每次只问一个问题，等用户回答后再问下一个。**

**Q1 核心场景**：用户最常执行的 1-3 个操作是什么？能否举一个完整的使用例子？

**Q2 呈现粒度**（涉及内容展示时提问，否则跳过）：信息是逐条/逐段标注，还是汇总展示？

**Q3 边界**：明确不做什么？

**Q4 涉及端**：哪些平台/客户端？后端服务？

**Q5 现有系统**：先建立架构事实，再提问：
  - 读取 `.doc/README.md`（如存在，列格式：`| 文件 | 覆盖范围 | 最后更新(YYYY-MM-DD) |`），根据覆盖范围判断需要精读哪几个文件，再定向读取；同时读取 `.doc/adr/README.md`（如存在，列格式：`| 编号(ADR-NNN) | 标题 | 状态 | 摘要 |`）
  - 不存在 README 时，才 glob 扫描对应目录
  - 整理架构事实表（新功能需要的能力 / 现有系统状态 / 结论），核查目的：避免为已有能力设计多余工程方案

**Q6 不确定性**：哪些工作结果不可预期？需先跑 Demo 验证；哪些有现成模式可直接集成？

判断标准：

| 判断问题 | 可预期（直接集成） | 不可预期（需 Demo） |
|---------|-----------------|-------------------|
| 有现成模式吗？ | 标准 REST CRUD、已有模式的 SSE | Prompt 效果、LLM 输出格式 |
| 结果可控吗？ | 输入-输出关系确定 | 依赖外部模型或不确定因素 |

Demo 命名规范：名称必须体现**能力**而非产品功能（判断标准：换一个产品，这个 Demo 还能用吗？）
- ❌ `hera-meeting-citation`（产品特定）
- ✅ `prompt-citation-stability`（通用能力）

完整模式额外追加：

**Q7 方案倾向**：是否已有备选技术方案？偏好快速验证还是长期迭代？

**Q7.5 竞品调研**（商业产品时执行，内部工具跳过）：
- 搜索竞品：使用 WebSearch，关键词组合 `"{产品类型} competitors"`、`"{核心功能} best tools {年份}"`、`"{目标用户} solutions"`
- 整理对比表：产品名称 x 核心功能维度，每格填写具体实现方式（禁止"快/慢/高/低"等单字评价，无法获取的信息标注"未公开"）
- 差异化分析：核心差异点 + 值得借鉴的设计 + 用户选择我们的理由
- 追问用户：相比这些竞品，我们的差异化在哪里？

**Q8 数据约束**：核心实体有哪些？数据量级？

## Step 2.5：检查已有 PRD

读取 `.doc/prd/README.md`（如存在），确认是否已有相同或高度相关的功能 PRD：

- **已有相关 PRD** → 定向读取该 PRD，判断是更新现有文档还是新建独立 PRD
- **无相关 PRD** → 继续新建

**判断锚点：新建 vs 更新**
- 正例（更新）：已有"AI 回复引用历史会议"，用户要补充"引用抽屉支持筛选"。结论：更新现有 PRD，不新建。
- 反例（新建）：已有"AI 回复引用历史会议"，用户要做"用户手动标注重点会议"。结论：两个功能独立，新建。
- 边界例（新建）：目标相同但技术方案完全不同。结论：新建 PRD 便于对比。

## Step 3：生成 PRD

### 3a. 脊梁提炼

写任何内容之前，先回答一个问题：

> **如果读者只记住一句话，应该是哪句？**

这句话就是文档的脊梁（Thesis）。

**要求**：
- 必须包含一个**判断**（"是/不是"、"应该/不应该"），不是描述（"我们要做 X"）
- ❌ "我们要做 About You 功能" — 描述，不是判断
- ✅ "About You 不是新功能，是视角切换——换两个 Prompt 就完事" — 有判断
- ❌ "优化会议摘要的引用体验" — 目标陈述，缺乏洞察
- ✅ "问题不在模型能力，在于我们把 context 喂错了" — 有因果判断

**脊梁贯穿规则**：核心洞察必须在文档中至少出现 3 次——开头立论、中段论证、结尾收束。不是重复同一句话，而是在不同章节中从不同角度强化同一个判断。

提炼完脊梁后，先展示给用户确认，再进入 3b。

### 3b. 论证链设计

围绕脊梁设计章节序列。核心规则：**每一章回答一个问题，这个问题由上一章自然引出。**

**参考模式**（选一种或组合，不是模板）：

| 模式 | 论证链 | 适用场景 |
|------|--------|---------|
| 视角切换型 | 旧视角 → 问题暴露 → 新视角（脊梁）→ 如何验证 → 如何落地 | 对现有能力重新定位 |
| 技术突破型 | 现状 → 瓶颈本质 → 突破点（脊梁）→ 方案设计 → 验证计划 | 技术升级或架构重构 |
| 问题解决型 | 痛点 → 根因分析 → 解法（脊梁）→ 方案对比 → 实施路径 | 最通用，解决明确问题 |

**章节数量由论证需要决定**，不由模板决定。可能 5 章，也可能 8 章。

**串联检验**：写完章节大纲后自检——读完第 N 章，读者是否自然产生一个问题，而第 N+1 章恰好回答？不是则调整章节顺序或合并。

将论证链大纲（章节标题 + 每章回答什么问题）展示给用户确认，再进入 3c。

### 3c. 选取论证积木

根据论证链需要选用积木，不是全部必填。

**产品模式积木池**：

| 积木 | 何时用 | 何时不用 |
|------|--------|---------|
| 痛点论述 | 需要让读者理解"为什么现在不行" | 问题已是共识，不需论证 |
| 方案论证 | 存在多个可行路径需要决断 | 只有一条明显正确的路 |
| 功能规格 | 需要明确交付物边界和迭代层次 | 纯探索性项目 |
| 核心场景 | 需要展示用户交互流程 | 纯后端/基础设施变更 |
| UI 与交互 | 涉及用户界面 | 无 UI 的后端/工具项目 |

**完整模式追加积木池**：

| 积木 | 何时用 | 何时不用 |
|------|--------|---------|
| 技术方案 | 需要说明实现路径 | 产品模式已足够 |
| 数据模型 | 需要新建或修改数据结构 | 复用现有结构 |
| 验证方案 | 方案可行性有不确定性 | 纯工程实施无技术风险 |
| 迭代路径 | 需要分阶段交付 | 一次性交付 |
| 边界与异常 | 需要处理失败路径 | 快乐路径已足够覆盖 |
| 竞品分析 | 商业产品 | 内部工具 |
| 双端落地 | 涉及 iOS/Android | 纯 Web/后端 |

**积木使用规则**：
- **同一观点只论述一次**：不在多个积木中重复。两处需要引用同一论点时，一处完整论述，另一处引用
- **合并不足独立的积木**：内容不足以支撑独立章节时，合并入相邻章节
- **服务论证链**：每个积木必须回答论证链中的某个问题。为了"完整性"而塞入的积木应删除
- **论证型写作**：采用"我观察到 X → 所以判断 Y → 因此决定 Z"的论证式，而不是"这是 A，这是 B，请看"的陈列式

**积木格式指引**（按需使用，非全部必填）：

- **功能规格**（涉及迭代时）：按 Layer 0（Demo 验证）→ Layer 1（必须）→ Layer 2（增强）分级，Layer 越高依赖越低不可倒置。Layer 0 需列 Demo 需求清单（`| Demo 名称 | 验证什么不确定性 | 对应 Layer 1 功能 |`）
- **方案论证**：维度 x 方案对比表，每格具体描述（禁止"快/慢"）；推荐用正向论证（"选 C 因为…"）+ 切换条件（"若 X 则改用方案 A"）
- **边界与异常**：按角色/层分组（AI 行为 / 客户端 / 后端），场景具体到触发条件，处理具体到行为（禁止"合理处理"）
- **双端落地**：每端一张映射表（`| 需求点 | 推荐改动文件 | 说明/验收 |`）
- **成功指标**：每条包含口径定义（分子/分母）+ 判定频率 + 达标线，禁止只写达标线不写口径

### 3d. 反模式门禁

写完全文后逐项自检，命中任何一项必须修正后再交付：

| 检查项 | 症状 | 处理 |
|--------|------|------|
| 脊梁缺失 | 核心洞察没有在 ≥3 处出现 | 在开头立论、中段论证、结尾收束中强化 |
| 凑数字 | 出现量化指标但无推导过程 | 补推导，或标注"待验证"并写明验证方法 |
| 机械 diff | "继承/替换/新增"式列表 | 改为"为什么保留 X"的判断论述 |
| 排除式论证 | "A 不行…B 不行…所以选 C" | 改为"选 C 因为…"的正向论证 |
| 零价值章节 | 删掉后文档完整性不受影响 | 删掉或合并入相邻章节 |
| 重复表达 | 两个表/段落换角度说同一件事 | 合并为一处，选最有力的角度 |
| 断裂感 | 读完第 N 章不知为什么要读第 N+1 章 | 调整章节顺序或补充过渡 |
| 陈列式写作 | 段落以"这是 X"罗列而非论证 | 重写为"因为 X → 所以 Y → 因此 Z" |

## Step 4：写入文件

路径：`.doc/prd/PRD-{kebab-case-title}.md`

`.doc/prd/` 不存在时先创建目录，同时创建 `.doc/prd/README.md`。

## Step 4.5：更新 `.doc/prd/README.md`

写入 PRD 文件后，同步维护索引：

```markdown
# PRD 索引

| 文件 | 功能摘要（一句话） | Demo 需求 | 状态 |
|------|-----------------|----------|------|
```

列格式定义见 `.doc/schema.md` §PRD。

**首次创建 README（兼容旧版本）**：glob 扫描 `PRD-*.md`，读取每个 PRD 的概述和 Demo 需求清单，生成完整索引表，状态默认为 `draft`。

**日常维护**：新建 PRD → 追加一行；更新 PRD → 修改对应行。

## Step 5：写入后提示

1. 显示生成的文件路径和模式（产品/完整）
2. 如有 Demo 需求清单：列出每项 Demo 名称，提示使用 `/ddemo {demo-名称}` 逐项生成落地方案
3. 所有 Demo 验证通过后，使用 `/dtask` 拆解集成任务
4. 产品模式下提示：如需补充技术方案细节，可使用 `/dprd full` 重新生成

## Step 6：交付前自检

```bash
rg -n '[""]' <prd-files>              # 智能引号
rg -n '/Users/|[A-Z]:\\' <prd-files>  # 绝对路径
rg -n $'\xef\xbf\xbd' <prd-files>     # 乱码字符（U+FFFD）
```

## 不做的事

- 不自动写入 dtask.json（任务拆解由 `/dtask` 负责）
- 不生成代码（PRD 是需求文档，不是实现）

## 写作门禁

- **ASCII 引号**：全文件仅允许 `"` (U+0022)，禁止 `"` `"` (U+201C/U+201D)
- **禁止绝对路径**：只写仓库相对路径或文件名
- **Demo 阶段禁止出现**（功能规格中存在 Layer 0 时适用）：数据库 DDL、多服务 API 契约设计、灰度发布策略、iOS/Android 完整代码片段
- **Demo 阶段必须包含**：Prompt v1 内容（LLM 类功能）、测试 case 矩阵（覆盖正常路径 + 边界）、量化通过标准
