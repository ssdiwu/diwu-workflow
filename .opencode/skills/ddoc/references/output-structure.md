# 输出结构选择指南

## 两种输出结构

### 领域驱动结构（推荐）

**适合**：有 3 个以上明确业务域、AI 是主要读者、长期维护的项目。

```
.doc/
├── README.md          # 文件一览 + 任务→域文件映射表
├── constraints.md     # 全局约束（跨域共享）
├── [domain-a].md      # 域A：数据/接口/业务规则/状态机
├── [domain-b].md      # 域B：同上
└── setup.md           # 环境域：技术栈、依赖、本地配置
```

**最小上下文加载原则**：`constraints.md` + 单个域文件 = 足够还原该域的完整功能。

**域文件设计约束**：
- **自包含**：该域的数据结构、接口、业务规则、状态机都在同一文件内
- **跨域引用显式化**：依赖其他域时，明确写出"见 `[domain].md §xxx`"
- **constraints.md 放全局约束**，不重复写入各域文件

### 分层结构

**适合**：业务域不明确的工具类产品、文档读者主要是人、小项目。

```
.doc/
├── data.md            # 数据层：实体、字段、关系
├── api.md             # 接口层：端点、入参、出参、错误码
├── logic.md           # 业务逻辑层：规则、状态机、算法
└── product.md         # 产品层：角色、旅程、权限、非功能需求
```

## 选择决策树

```
有 3 个以上明确业务域？
  ├─ 是 → 领域驱动结构
  └─ 否 → 业务域不明确？
      ├─ 是 → 分层结构
      └─ 否 → 项目规模小？
          ├─ 是 → 分层结构
          └─ 否 → 领域驱动结构
```

## 领域驱动结构详解

### 何时使用

- 产品有明确的业务域划分（如：认证域、支付域、订单域）
- AI 是主要读者（需要快速加载最小上下文）
- 项目长期维护，需要模块化管理

### 域文件模板

```markdown
# [域名] 域

## 数据层

### 实体
[实体列表 + 字段定义]

### 关系
[erDiagram]

## 接口层

### API 端点
[端点列表 + 入参/出参]

## 业务逻辑层

### 业务规则
[核心规则列表]

### 状态机
[stateDiagram-v2]

### 关键算法
[算法描述]

## 跨域依赖

- 依赖 [domain-b].md §xxx
```

## 分层结构详解

### 何时使用

- 业务域不明确（工具类产品、基础设施项目）
- 文档读者主要是人（需要按技术层次理解）
- 小项目（< 10 个核心实体）

### 层文件说明

- **data.md**：所有数据实体、字段、关系
- **api.md**：所有 API 端点、入参、出参、错误码
- **logic.md**：所有业务规则、状态机、算法
- **product.md**：用户角色、旅程、权限、非功能需求
