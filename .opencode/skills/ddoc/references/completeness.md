# 两层完整性检查详解

## 概述

完整性检查分两层：
1. **第一层：结构检查**（有没有？）
2. **第二层：深度检查**（够不够？）

不是 4×5=20 个格子每个都必须填满。有意识地检查过，标注"不适用"即可。

## 第一层：结构检查（有没有？）

| 层 | 检验问题 | 检查项 |
|----|---------|-------|
| **数据层** | 能写出完整的数据库 migration 吗？ | 实体列表、字段定义（名/类型/约束）、实体关系、查询模式 |
| **接口层** | 能写出完整的 API client 吗？ | 端点列表、入参格式、出参格式、认证方式、错误码 |
| **业务逻辑层** | 不看代码就能实现核心规则吗？ | 业务规则、状态流转（状态机+转移表）、关键算法、边界条件、可视化图表 |
| **产品层** | 新人能理解"这个产品是做什么的"吗？ | 用户角色、核心用户旅程、权限矩阵、非功能需求 |

### 数据层检查项

- **实体列表**：所有核心实体都列出了吗？
- **字段定义**：每个实体的字段（名称/类型/约束）都写了吗？
- **实体关系**：实体之间的关系（1:1、1:N、N:M）画了吗？
- **查询模式**：常见查询场景列出了吗？

### 接口层检查项

- **端点列表**：所有 API 端点（路径 + HTTP 方法）都列出了吗？
- **入参格式**：每个端点的入参结构写了吗？
- **出参格式**：每个端点的出参结构写了吗？
- **认证方式**：认证机制（JWT/OAuth/API Key）写了吗？
- **错误码**：错误码列表和含义写了吗？

### 业务逻辑层检查项

- **业务规则**：核心业务规则列出了吗？
- **状态流转**：状态机 + 转移表画了吗？
- **关键算法**：核心算法逻辑写了吗？
- **边界条件**：异常情况和边界条件写了吗？
- **可视化图表**：flowchart、stateDiagram-v2、erDiagram 画了吗？

### 产品层检查项

- **用户角色**：所有用户角色列出了吗？
- **核心用户旅程**：主要用户流程写了吗？
- **权限矩阵**：角色权限表写了吗？
- **非功能需求**：性能、安全、可用性要求写了吗？

## 第二层：深度检查（够不够？）

对第一层已覆盖的每一项，用五约束维度逐一检验：

### 业务约束检查

- 不变的规则写了吗？
- 特别是"不能做什么"（负面约束）写了吗？
- 这些规则 100 年后还成立吗？

### 时序约束检查

- 状态转移表画了吗？
- "先后顺序"写清了吗？
- 非法转移的处理策略写了吗？

### 跨端约束检查

- 多端相关的概念有唯一定义吗？
- 枚举值在所有端完全一致吗？
- 数据格式在所有端统一了吗？

### 并发约束检查

- 两个用户/线程同时操作的策略写了吗？
- 锁机制（乐观锁/悲观锁）写了吗？
- 并发冲突的解决策略写了吗？

### 感知约束检查

- 响应时间有具体阈值吗？
- 加载策略写了吗？
- 用户可感知的性能指标写了吗？

## 检查清单

### AI1 自审清单（逆向模式）

□ 每个数据实体是否有完整字段列表？
□ 每个 API 是否有入参和出参的具体格式？
□ 每条业务规则是否有对应的异常情况处理？
□ 每个用户流程是否覆盖了"失败路径"？
□ 是否有任何地方用了"等"、"类似"、"根据情况"这类模糊表达？
□ 所有有状态的实体是否都画了 stateDiagram-v2？
□ 并发场景是否被考虑？
□ [待确认] 标记是否只在真正无法确认时使用？
□ 核心业务流程是否有 flowchart？
□ 数据实体关系是否有 erDiagram？

对每个"否"，直接补充内容后再交付。
