# 架构约束

> **规则约束级别说明**：本文件定义架构约束的核心规则。除非特别标注 `[建议]`，否则都是必须遵守的约束。

## Constraints

| Dimension    | Constraint                                                                                       | Verification                                                     |
|--------------|--------------------------------------------------------------------------------------------------|------------------------------------------------------------------|
| Business     | 命令/接口描述层与实现逻辑层分离；描述文件不包含可执行代码                                            | 描述文件内无可执行代码，仅有行为指令或接口定义                      |
| Temporal     | 功能变更时版本号同步更新；遵循语义化版本规范（X.Y.Z）                                              | 功能变更 commit 包含版本号更新；格式符合 semver                     |
| Cross-platform | 路径引用兼容目标平台；禁止使用平台特定分隔符                                                     | 路径中无平台特定分隔符                                             |
| Concurrency  | 明确并发策略；静态资源无需并发控制时声明                                                          | 并发场景有明确策略或无并发声明                                      |
| Perception   | 关键路径可观测；降级行为用户可感知                                                                | 降级时有明确报错或警告，不静默失败                                  |

## Constraints 判断锚点

> **规则约束级别说明**：本章节为 Constraints 表中的每条约束提供现象→判断→动作框架的判断锚点，正例/反例/边界例矩阵作为判断环节的排除项。

---

### Business 约束：描述与实现分离

**规则来源**：Constraints 表 Business 行

**框架映射**：
- 现象 = 描述文件中出现了什么代码形态
- 判断 = 这是实现逻辑还是描述性引用
- 动作 = 保留描述还是分离实现

**正例**：命令文档仅包含指令格式说明和参数定义，无任何条件逻辑或脚本代码。结论：合规，无需修改。

**反例**：描述文件中直接嵌入 `if [ -d "$cache" ]; then ...` 类的条件逻辑，或引用具体执行脚本作为默认步骤。结论：实现逻辑混入，必须分离到实现层，描述文件只保留行为定义。

**边界例**：描述文档中使用「参考 XXX」形式引用知识文件（非执行脚本）。结论：引用的是静态知识而非执行逻辑，合规。

---

### Degradation Paths 约束：明确报错不静默失败

**规则来源**：Degradation Paths 表 + exceptions.md §BLOCKED 判定

**框架映射**：
- 现象 = 依赖出现了什么异常状态
- 判断 = 这属于哪级降级、用户是否能感知
- 动作 = 报错、警告还是静默降级

**正例**：依赖不可用时明确报错并给出恢复指引，退出码非零。结论：明确报错，合规。

**反例**：依赖不可用时静默退出，返回码 0，无任何错误输出。结论：静默失败，违规。用户无法判断是成功还是未执行。

**边界例**：非关键依赖不可用时打印警告但核心功能仍正常完成（exit 0）。结论：非关键依赖降级，警告用户但不完全阻塞，合规。

## Boundaries

| Type     | Internal                               | External                         | Crossing Method          |
|----------|----------------------------------------|----------------------------------|-------------------------|
| Domain   | 核心业务模块                            | 外部系统 / 第三方服务             | API / 接口契约            |
| Platform | 应用自身组件                            | 运行环境 / 宿主平台               | 配置 / 适配层             |
| Data     | 源数据 / 模板文件                        | 生成产物 / 用户数据              | 复制 / 转换流程           |

## Degradation Paths

| Dependency          | Failure                          | Fallback                               | User Perception              |
|---------------------|----------------------------------|----------------------------------------|----------------------------|
| 核心依赖             | 不可用                           | 明确报错 + 恢复指引                     | 明确报错，不静默失败          |
| 非关键依赖           | 不可用                           | 打印警告 + 降级运行                     | 警告通知，功能部分可用        |

## 五维约束设计参考

基于产品特性，在设计阶段识别以下五个维度的约束：

| 维度 | 判断问题 | 例子 |
|-----|---------|------|
| **业务约束** | 100 年后还成立吗？ | 「原始录音不可变，摘要可重新生成」 |
| **时序约束** | 有没有执行路径可以绕过这个规则？ | 「停止录音后同一 session 不可恢复」 |
| **跨端约束** | 如果两个端定义不同，这是 bug 吗？ | 「CardType 在所有端完全一致」 |
| **并发约束** | 两个线程同时做这件事会出问题吗？ | 「状态机事件串行处理」 |
| **感知约束** | 超过这个阈值，用户会注意到吗？ | 「音频源切换 < 100ms」 |

使用方式：新增功能模块时，逐行过上表，把命中的约束写入本文件的 Constraints 表。
