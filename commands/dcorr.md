# /dcorr — 纠偏恢复

> **定位**：偏航时的恢复协议，不是开工前的完整规范，也不是全程运行总规范。
> **目标**：把"继续补说明硬推"改成"先回到主线，再继续执行"。

## 触发条件

当出现以下任一情况时，停止扩写，启动本命令：

- 同一问题已经被纠偏两次以上，仍沿旧路径推进
- 同一个前提被反复解释
- 输出越来越像通用套路，越来越不像当前任务
- 讨论越来越长，但下一步动作越来越模糊
- 已经改动，却拿不出运行态或输出层证据
- 开始同时引用多个目录、多个入口、多个"真相源"

## Step 1：停止当前动作

偏航时，先停掉这 6 类动作：

| # | 停止项 | 为什么停 |
|---|--------|---------|
| 1 | 沿已被否定的路径继续扩写 | 路径已否定，继续只会扩大误差 |
| 2 | 把副本项目、旧入口、旧文档当当前主线 | 副本不是 canonical（规范真相源） |
| 3 | 把界面表象直接当主链根因 | 表象≠根因，需调用链证据支撑 |
| 4 | 把"测试红了"或"提示成功"直接当最终结论 | 需确认底层状态是否真的切换 |
| 5 | 把问题直接归因给模型、提示词或代码其中一层 | 归因前先排除环境/进程/端口等基础项 |
| 6 | 在没有证据的情况下宣称"已经完成" | 未验证=未完成 |

**判断依据**：只要命中上述任一项，必须先进入 Step 2 四行重写，不得跳过。

---

## Step 2：四行重写

纠偏时，先用下面四行重新写当前任务：

```text
当前目标：
- 这次到底要得到什么结果

当前主线：
- 唯一主线目录
- 唯一运行入口或工作入口
- 唯一 canonical（规范真相源）

当前现象：
- 已观测到的事实

当前缺口：
- 缺什么证据，导致还不能继续推进
```

**判断依据**：
- 如果四行写不出来 → 说明还没回到主线，不要继续推进
- 如果"唯一主线目录"写了多个 → 说明主线不唯一，需先收敛
- 如果"当前缺口"写的是推测而非具体证据 → 回到现象层补事实

**轻量变体**（纯写作/低运行态任务）：

```text
当前目标：
- 这次到底要产出什么

当前主线：
- 唯一主线目录
- 唯一运行入口或工作入口
- 唯一 canonical

当前现象：
- 已出现的事实、偏题点或结构失真点

当前动作：
- 下一步只做什么
- 用什么证据判断已经回到主线
```

## Step 3：误判排查

很多偏航不是"不会做"，而是第一判断先错了。看到下面现象时，优先排除对应误判：

| 现象 | 不要先说 | 默认先查 |
|------|---------|---------|
| 今天突然不通、同提交表现不一致 | 代码回归 | 运行进程、端口、代理、环境变量、依赖服务 |
| 没命中、结果空、候选缺失 | 实现失效 | 样本、权限、账号、数据存量 |
| 页面白屏、错位、打不开 | 后端主链挂了 | 前端读层、渲染层、开发服务 |
| `tool-loop-guard`、空查询、错工具 | 模型不行 | 路由、参数、护栏、接口契约 |
| 脚本失败、测试失败、`200`、成功提示 | 真实失败 / 真实成功 | 底层状态是否真的切换 |
| 结构化结果差 | 提示词一定错 / 代码一定错 | 输入层、提示词层、后置判定层、回退层 |

**判断依据**：
- 先查表中右侧列的排查项，排除后再考虑左侧列
- 排查顺序：从外到内（环境→配置→接口→实现）

---

## Step 4：入口重判

纠偏后，不要默认继续主线。先重新判断当前任务该怎么进入。

### 门控 A：可以直接继续

同时满足三项时，可直接继续做：

- 改动范围小
- 结果可预期
- 预期差异可以一句话说清

### 门控 B：必须先写最小规格

满足任一条件时，先写最小规格：

- 结果不能一句话说清预期差异
- 依赖外部系统、运行时链路或跨层集成
- 输出需要稳定格式、稳定字段或稳定验收
- 需要交接给其他人或下一轮继续推进

最小规格模板：

```text
目标：
- 这次到底要得到什么结果

输入：
- 已知材料
- 关键约束
- 外部依赖

输出：
- 最终产物
- 放在哪里
- 返回形式

验收标准：
- 看什么证据算完成
- 哪些边界不能破
- 哪些项仍待验证
```

### 门控 C：更适合先探索验证

满足任一条件时，先做最小验证：

- 外部依赖多且不可控
- 真实问题落点不清楚
- 组合行为不稳定
- 一旦方向错了，回滚成本高

**判断依据**：三个门控互斥，按 A → B → C 顺序评估，命中即止。

---

## Step 5：恢复执行

重新开始推进时，第一条输出必须使用这个骨架：

```text
现象：
- 我现在看到的事实是什么

判断：
- 基于这些事实，我认为当前问题更像什么
- 哪些误判已经排除

动作：
- 下一步具体做什么
- 怎样验证
```

**禁止行为**：恢复后的第一条输出里不得宣称"已经搞定"。必须先给出现象和判断依据。

### 判断必须有依据

涉及判断、归纳、取舍、方案选择时，至少补足一种依据：

- 正例 / 反例 / 边界例
- 已观测到的异常现象
- 已验证的约束
- 已确认的不做项
- 数据来源 / 时间范围 / 计算方式

如果连最小依据都给不出来，不要继续恢复主线，应先补事实。

### 验证优先级

完成前，优先找高等级证据：

| 等级 | 说明 | 行为 |
|------|------|------|
| Level 1 | 运行态和真实输出已发生可识别变化 | 可宣称完成 |
| Level 2 | 调用链命中证据、底层状态切换证据 | 可宣称完成 |
| Level 3 | 自动化测试、最小验证脚本、断言 | 可宣称完成 |
| Level 4 | 日志、截图、页面现象 | 仅辅助说明 |
| Level 5 | 代码 diff、口头说明 | 不可宣称完成 |

默认优先找 Level 1-3，再用 Level 4-5 辅助。只有 Level 5 时不应宣称已完成。

## 与 InProgress 任务共存时的行为

当 `/dcorr` 在有 InProgress 任务的环境中触发时：

1. **不改变 task.json 状态**：纠偏是过程修正，不是状态转移。InProgress 任务保持 InProgress。
2. **记录到 session 文件**：将纠偏过程（触发原因、四行重写结果、误判排除结论）写入当前 session 文件。
3. **与 Stop hook checkpoint 的关系**：Stop hook 的 checkpoint 机制检测的是 InProgress 任务是否存在，`/dcorr` 不影响该检测。纠偏完成后正常继续实施，Stop hook 仍会按 continuous_mode 配置决定是否续跑。
4. **恢复失败处理**：若经五步仍无法回到主线，按止损顺序操作：
   - 切掉已污染的历史上下文
   - 将任务缩到最小可执行单元
   - 明确写出缺的边界、证据或外部依赖
   - 请求人类补充唯一主线、验收边界或判别依据

## 结束前的最小检查

纠偏恢复并继续实施后，结束前至少回答四个问题：

1. 我验证的是哪一层：文档、代码、配置、调用链、运行态还是输出？
2. 是否真的走到了新链路？
3. 结果是否出现了可识别变化？
4. 哪些部分已经验证，哪些部分还未验证？

无法验证时应明确写出：未验证项、原因、后续验证方式。

## 不要退化成什么

- 不要退化成"全程运行总规范"——那是融合版承担的范围
- 不要退化成"所有任务都按排障处理"的手册
- 不要退化成"只会让 AI 停下，却不会让 AI 重新起步"的禁止清单
- 不要退化成"继续堆长说明"的另一种形式
