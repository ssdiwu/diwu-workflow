# ddoc 快速执行指南

## 快速决策树

```
用户说"写文档" → 问：有代码还是有需求？
  ├─ 有代码 → 逆向模式
  └─ 有需求 → 正向模式
```

## 正向模式核心步骤

1. **找约束**：用五约束维度（业务/时序/跨端/并发/感知）从需求中提取约束
2. **定义原子**：识别不可再拆的最小单元（实体/状态/变换/事件）
3. **设计组合**：确定组合方式（结构/管道/状态机）
4. **划定边界**：确定领域边界、平台边界、数据边界
5. **设计降级**：列出依赖链，定义失败时的降级路径
6. **设计同步**：确定 Source of Truth 和冲突策略

详细步骤见 references/forward-mode.md

## 逆向模式核心步骤

1. **AI1 写文档**：读代码库，按四层结构（数据/接口/业务逻辑/产品）写文档，标注代码来源
2. **AI1 自审**：用自审清单检查完整性（10 项检查）
3. **AI2 gap 分析**：结构化审查，输出缺口表
4. **逆向 Spec 验证**：对关键模块尝试写代码，发现真实缺口
5. **AI1 补充**：根据缺口补充文档
6. **AI2 终审**：确认无缺口

详细步骤见 references/reverse-mode.md

## 两层完整性检查

### 第一层：结构检查（有没有？）

- 数据层：实体列表、字段定义、实体关系、查询模式
- 接口层：端点列表、入参格式、出参格式、认证方式、错误码
- 业务逻辑层：业务规则、状态流转、关键算法、边界条件、可视化图表
- 产品层：用户角色、核心用户旅程、权限矩阵、非功能需求

### 第二层：深度检查（够不够？）

对第一层已覆盖的每一项，用五约束维度检验：
- 业务约束：不变的规则写了吗？特别是"不能做什么"
- 时序约束：状态转移表画了吗？
- 跨端约束：多端相关的概念有唯一定义吗？
- 并发约束：两个用户/线程同时操作的策略写了吗？
- 感知约束：响应时间、加载策略有具体阈值吗？

详细说明见 references/completeness.md

## 常见问题

**Q: 逆向模式时代码太大怎么办？**
A: 按映射优先级扫描：数据模型 > API 路由 > 状态机 > 业务服务 > 配置文件 > 测试文件

**Q: 正向模式时需求不清晰怎么办？**
A: 每次只问一个问题，理解目的、约束、成功标准；提出 2-3 个方案并给出推荐

**Q: 文档写完后如何验证完整性？**
A: 用两层完整性检查：先结构检查（有没有），再深度检查（够不够）

**Q: 如何选择输出结构？**
A: 有 3 个以上业务域 → 领域驱动；工具类产品或小项目 → 分层

**Q: 逆向模式时遇到无法确认的内容怎么办？**
A: 标注 `[待确认]`，不编造。`[待确认]` 标记不应超过总条目的 10%
