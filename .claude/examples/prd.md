# PRD 示例

## 正例

### 示例 1：用户注册功能
```
1. 功能概述
用户通过邮箱注册账号，系统发送验证码完成验证。

2. 业务流程
2.1 用户填写注册表单（邮箱、密码、昵称）
2.2 系统校验邮箱格式和密码强度
2.3 系统发送 6 位数字验证码到用户邮箱
2.4 用户输入验证码完成注册

3. 约束条件
3.1 验证码有效期 10 分钟
3.2 同一邮箱 1 分钟内只能发送 1 次验证码
3.3 密码长度 8-20 位，必须包含字母和数字
```

### 示例 2：订单支付功能
```
1. 功能目标
用户选择商品后通过第三方支付完成付款。

2. 支付流程
2.1 用户在购物车确认商品和金额
2.2 用户选择支付方式（微信/支付宝）
2.3 系统生成订单并跳转到支付页面
2.4 用户完成支付后系统接收回调
2.5 系统更新订单状态并发送确认通知

3. 异常处理
3.1 支付超时（30 分钟）自动取消订单
3.2 支付失败时保留订单，用户可重新支付
3.3 重复支付时系统自动退款
```

## 反例

### 反例 1：技术实现混入业务描述
```
1. 用户注册功能
使用 bcrypt 加密密码，存储到 PostgreSQL users 表，通过 nodemailer 发送验证码。
```
**问题**：PRD 应描述"做什么"，不应包含"怎么做"的技术细节。

### 反例 2：序号层级混乱
```
1. 用户注册
1.1 填写表单
2. 发送验证码
2.1 校验邮箱
1.2 输入验证码
```
**问题**：序号跳跃（1.1 → 2 → 2.1 → 1.2），层级关系不清晰。

### 反例 3：中英文混用
```
1. User Registration
用户通过 email 注册账号，系统 send verification code。
```
**问题**：违反纯中文原则，技术术语应保持英文但句子结构应为中文。

## 边界例

### 边界例 1：技术术语保留英文
```
1. API 接口设计
系统提供 RESTful API，支持 OAuth 2.0 认证。
```
**判断**：技术术语（API、RESTful、OAuth）保留英文，符合规范。

### 边界例 2：引用外部系统
```
2. 支付流程
2.1 调用微信支付 API 生成预付单
2.2 用户在微信客户端完成支付
```
**判断**：描述与外部系统的交互，属于业务流程，可接受。

### 边界例 3：性能指标
```
3. 性能要求
3.1 接口响应时间 < 200ms
3.2 支持 1000 并发用户
```
**判断**：性能指标属于约束条件，应放在独立章节，不与业务流程混合。
